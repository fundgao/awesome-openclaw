### **Skills 的本质：不是文件，是文件夹**

对 Skills 最常见的误解是“它就是个 Markdown 文件”。实际上，**一个 Skill 是一个文件夹**，可以包含：

*   **脚本**（Python、Shell、JS）
*   **参考资料**（代码片段、API 文档）
*   **配置文件**（config.json）
*   **数据资产**（模板、示例输出）
*   **记忆存储**（日志文件、SQLite 数据库）

Claude Code 能够发现、探索并操作这整个文件夹结构。这意味着你可以把整个工具链打包进一个 Skill，让 Claude 像使用一个智能工具箱一样去使用它，而不只是读一段文字说明。

---

### **Skills vs MCP：大脑和手脚的区别**

很多人搞不清 Skills 和 MCP 的边界，认为两者都是“扩展 Claude 能力”的工具。一个生动的比喻能帮你厘清：

> **你雇了一位新员工。给他配发工作账号（GitHub、Slack、数据库访问），这是 MCP，是“手脚”，让他能操作外部系统。但仅有账号不够，你还要给他一本操作手册，说明公司流程、哪些事要怎么做、哪些坑不能踩，这是 Skills，是“大脑”，是标准作业程序（SOP）。**

两者的核心差异：

*   **MCP 是手脚**：本质是**工具连接**，解决 **“能做什么”**。例如 GitHub MCP、Slack MCP、数据库连接，它们全局生效，持续运行。
*   **Skills 是大脑**：本质是**流程知识**，解决 **“到了那里该怎么做”**。例如代码审查 SOP、部署流程、内部规范，它们按需激活，针对性强。

**MCP 解决的是“连接到哪里”的问题，Skills 解决的是“连接后该怎么做”的问题。** 两者组合，是能力和流程的相乘，而非简单相加。

### **为什么需要 Skills？三个核心瓶颈**

没有 Skills，AI 辅助开发通常会卡在三个地方：

1.  **上下文盲区**：Claude 不了解你的内部库、私有 API、团队规范。每次都要重新解释，且容易理解偏差。
2.  **流程真空**：Claude 擅长单点任务，但复杂工作流涉及多步骤、多工具、多决策节点。没有 Skills，Claude 只能即兴发挥，输出质量不稳定。
3.  **执行焦虑**：涉及高风险操作（如数据库变更、生产部署），你不敢放手让 Claude 自主执行，因为不确定它是否会遗漏关键步骤。

Skills 针对这三个瓶颈有精准的解法：**参考技能**消灭上下文盲区，**工作流技能**和**自动化技能**填补流程真空，**验证技能**和**动态钩子**解除执行焦虑。

### **9 种 Skills 类型**

Anthropic 整理内部数百个 Skills 后，发现它们自然地聚集为 9 个类别。清晰归类是写出好 Skill 的第一步。

| 类型 | 用途 | 核心价值 | 示例 |
| :--- | :--- | :--- | :--- |
| **1. 参考技能** | 教 Claude 如何正确使用内部库、CLI 或 SDK。 | 突破 Claude 的默认认知盲区。 | `billing-lib`（内部计费库的雷区清单）、`frontend-design`（提升设计品味） |
| **2. 验证技能** | 告诉 Claude 如何测试和验证代码是否真的跑通了。 | 闭环验证，让 Claude 对结果负责。 | `signup-flow-driver`（无头浏览器跑通注册全流程）、`checkout-verifier`（用测试卡驱动结账） |
| **3. 数据技能** | 连接数据和监控系统，让 Claude 能直接查询生产数据。 | 让 Claude 变成真正懂你业务数据的分析师。 | `funnel-query`（包含实际表名的查询）、`grafana`（问题到仪表盘的映射表） |
| **4. 自动化技能** | 把重复的日常工作流变成一条命令。 | 把每天重复三遍的操作压缩成零成本。 | `standup-post`（汇总生成日会报告）、`weekly-recap`（一键生成周报） |
| **5. 脚手架技能** | 为代码库中的特定功能生成框架模板。 | 统一工程规范，让新人上手速度翻倍。 | `new-migration`（迁移文件模板加常见坑提醒）、`create-app`（预设认证和日志的新应用） |
| **6. 审核技能** | 在组织内执行代码质量控制，可集成到 Git Hook 或 CI。 | 把代码审查标准系统化。 | `adversarial-review`（多维度并发审查代码）、`code-style`（强制执行特定代码风格） |
| **7. 工作流技能** | 涵盖获取、推送和部署代码的端到端流程。 | 把部署变成无聊的例行任务。 | `babysit-pr`（监控 PR、重试 CI、解决冲突）、`deploy-service`（带自动回滚的发布流程） |
| **8. 调查技能** | 接受症状（如报障、告警），跨工具调查并生成结构化报告。 | On-call 时不用自己一个系统一个系统去翻。 | `service-debugging`（将症状映射到查询模式）、`log-correlator`（根据请求 ID 聚合日志） |
| **9. 维护技能** | 执行例行运维任务，通常包含破坏性操作和保护措施。 | 高风险操作标准化，减少人为失误。 | `resource-orphans`（清理孤儿资源）、`cost-investigation`（分析云成本异常原因） |

### **一个 Skill 的内部结构**

每个 Skill 的核心文件是 `SKILL.md`，顶部是 YAML frontmatter，下面是内容主体。

**1. YAML frontmatter 四个关键字段：**
```yaml
---
name: deploy-service
description: 当需要部署服务到生产环境时触发。覆盖构建、冒烟测试、流量切换、回滚全流程。
disable-model-invocation: false
allowed-tools:
  - Bash
  - Read
  - Write
---
```
*   `description` 极其重要，它不是摘要，而是**触发条件**。Claude 会扫描所有 Skills 的 description 来判断何时该用哪个 Skill。
*   `disable-model-invocation: true` 适用于高危 Skill，强制人工确认后才执行。
*   `allowed-tools` 限制该 Skill 能使用的工具范围，对控制权限边界很有用。

**2. 内容主体通常分为两种：**
*   **参考内容（Reference）**：背景知识，如库的边缘情况、内部规范、常见误区。
*   **任务内容（Task）**：操作指南，如具体流程、决策树、验收标准。

**3. 一个实用原则**：`SKILL.md` 保持在 500 行以内。超出的内容拆分到子文件夹（如 `references/`、`assets/`、`scripts/`），在主文件中用相对路径引用。文件系统本身就是你的渐进式提示机制，Claude 会在需要时去读取对应文件。

### **制作优质 Skills 的 10 个秘诀**

1.  **聚焦高信号上下文**：专注于“Claude 不知道的部分”——你们的内部规范、历史决策、特有工具的奇怪行为。
2.  **捕捉“坑点”**：Skill 中价值最高的内容是“踩过的坑”。每次 Claude 失败，就把这个坑写进 Skill。
3.  **用好整个文件系统**：用文件夹结构来组织内容，实现渐进式揭示。
4.  **指令要灵活，不要太具体**：提供所需的信息框架，让 Claude 能灵活处理不同情况。
5.  **用 `config.json` 处理交互配置**：让用户一次性配置好参数（如目标 Slack 频道），避免每次重复询问。
6.  **让 Skills 有记忆**：用追加日志文件或 SQLite 数据库让 Skills 记住历史状态。使用 `${CLAUDE_PLUGIN_DATA}` 作为持久化存储路径。
7.  **直接给 Claude 代码**：把可复用的函数、辅助库放进 Skill 文件夹，让 Claude 专注于如何组合它们解决问题。
8.  **动态注入运行时上下文**：在 `SKILL.md` 中使用 `!` 命令，让 Claude 每次执行时都获取最新的 git 历史、环境变量等信息。`$ARGUMENTS` 变量可接收调用时传入的参数。
9.  **用 `context: fork` 启动子代理**：在 Skill 内启动多个独立的子代理并行处理任务（如多维度代码审查），效率更高，视角更纯粹。
10. **用动态钩子控制行为边界**：创建只在被调用时激活的钩子（如 `/careful` 阻止危险命令），需要时开启，不需要时关闭，避免全局限制让 Claude 束手束脚。

### **真实案例：代码审查 Skill 是怎么工作的**

1.  **触发**：用户执行 `/cr` 或说“帮我审查这个 PR”。
2.  **第一阶段：评估规模**：脚本统计改动行数。小于200行自动审查；200-500行提示建议拆分；超过500行建议分批审查。这决定了后续审查的深度。
3.  **第二阶段：自动化检查**：先跑 lint、类型检查、单元测试等确定性工具，把能自动找出的问题交给工具。
4.  **第三阶段：多维度并发审查**：通过 `context: fork` 启动三个并发子代理，分别负责**架构与设计**、**安全性**、**性能**的审查。
5.  **第四阶段：生成结构化报告**：汇总成统一格式的报告，问题按严重程度分级（阻断、建议、细节），每条附带代码位置、原因和改进建议。

整个流程下来，一个原本需要高级工程师30-45分钟的PR审查，被压缩到几分钟，且覆盖维度更全面。

### **怎么开始？**

如果你刚开始使用 Skills，建议按这个优先级来：

*   **第一步**：识别你团队最痛的重复工作，从「自动化技能」入手，效果最直接。
*   **第二步**：找出 Claude 最常犯错的地方，建一个「参考技能」，把坑点列进去。
*   **第三步**：为核心业务流程建「验证技能」，让 Claude 能自我检验输出是否正确。
*   **之后**，结合业务需要逐步扩展到其他类型。

### **写在最后**

Skills 的本质，是把团队的集体经验编码成 AI 可执行的流程知识。每个踩过的坑、每套标准化流程、每条内部规范，都可以写进 Skills，成为 Claude 永久携带的“肌肉记忆”。

有一个公式值得记住：**能力 × 流程 = 真正的 AI 生产力**。

MCP 给了 Claude 访问工具的能力，但能力本身不等于效率。Skills 把你的团队流程注入进去，才完成了这个乘法。随着时间积累，你的 Skills 库会成为团队最有价值的 AI 资产，因为它沉淀了你们团队独有的、任何通用 AI 都无法替代的经验。

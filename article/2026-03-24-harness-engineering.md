# 一文看懂 Harness engineering：智能体时代的 AI 编程驾驭之道

原文链接：https://mp.weixin.qq.com/s/YYurQM9EUuyshuW20YAMJQ

## 一句话总结

这篇文章的核心观点是：**在 Agent 编程时代，决定效果上限的往往不只是模型本身，而是模型之外那整套“Harness（驾驭系统）”**——也就是状态、工具、环境、反馈回路、约束与执行编排。

## 核心结论

作者借 LangChain 文章《The Anatomy of an Agent Harness》提出一个非常抓重点的判断：

> **Agent = Model + Harness**

换句话说，如果不是模型本身，那几乎都属于 Harness。

文章的重点不在提示词怎么写，而是在强调：**真正让 AI 变成可持续干活的 Agent，不是单次推理能力，而是外围工程系统。**

## 什么是 Harness

Harness 可以理解为：**模型之外的一整套执行外壳和操作系统**。

它通常包括：

- 系统提示词：定义角色、边界和目标
- 工具 / Skills / MCP：给模型扩展外部能力
- 基础设施：文件系统、沙箱、浏览器等运行环境
- 编排逻辑：子 Agent、任务拆分、模型路由
- 钩子与中间件：压缩、续写、代码检查、确定性流程

文章认为，用“模型 vs Harness”来划分系统边界，比泛泛谈 Agent 更清楚，因为它会逼你回答两个问题：

1. 模型本身到底负责什么？
2. 剩余能力应该由系统补什么？

## 为什么需要 Harness

原因很简单：**很多我们希望 Agent 具备的能力，模型本身根本没有。**

模型本质上只是一个输入 → 输出函数：

- 输入：文本 / 图像 / 音频
- 输出：文本

它并不会天然拥有这些能力：

- 持续记住多轮状态
- 执行代码
- 获取实时信息
- 操作真实环境
- 长时间稳定推进任务

这些能力都需要由模型外部补齐，也就是由 Harness 提供。

所以 Harness Engineering 的核心思路不是“再调一下 prompt”，而是：

- 先定义你想让 Agent 做成什么样
- 再把缺的能力一项项补到系统里

## 文章拆解的几个关键 Harness 组件

### 1. 文件系统：状态持久化与上下文卸载的基础

作者认为，文件系统不是附加功能，而是 Harness 的基础原语之一。

因为模型上下文窗口很有限，没有文件系统时，所有信息都只能靠人不断复制粘贴喂进去，这既低效也不利于自主执行。

有了文件系统之后，Agent 才真正具备这些能力：

- 拥有自己的工作空间
- 按需加载信息，而不是全部塞进上下文
- 把中间结果落盘
- 跨会话保留状态
- 让人类和多个 Agent 围绕同一份文件协作

如果再加上 Git，能力会进一步升级：

- 记录每一步改动
- 出问题时回滚
- 开分支做不同尝试

文章的潜台词很明确：**没有文件系统，Agent 很难从“会回复”升级成“会做事”。**

### 2. Bash + 代码执行：给 Agent 一台能干活的机器

如果你给模型的只是固定工具列表，那它只能在你预设的边界里行动。

问题在于，现实任务无法被预先穷举成有限工具。

因此，更强的方式不是提供几十个固定工具，而是直接给 Agent 一个通用执行环境：

- Bash
- 代码执行能力
- 可组合的 CLI 工具

这样带来的变化是：

- 模型可以自己写脚本解决问题
- 可以临时“造工具”
- 可以组合已有能力拼工作流

文章这里强调了一个思路转变：

- 不是“设计尽可能完整的工具列表”
- 而是“给它一台能工作的机器”

这其实就是很多现代编码 Agent 越来越像终端助手的原因。

### 3. 沙箱环境与工具链：安全执行 + 自我验证

给 Agent 能执行代码还不够，还得给它一个**可以放心折腾的环境**。

直接在本地跑模型生成的代码存在明显风险，而且也不利于并发与隔离。

所以文章强调，Harness 需要提供沙箱环境，主要解决两件事：

#### 安全性

- 隔离执行环境
- 限制命令与权限
- 控制网络访问
- 出错也只会限制在沙箱内部

#### 可扩展性

- 可按需创建环境
- 多任务并行执行
- 用完即可销毁，避免状态污染

此外，沙箱只是底座，真正让 Agent 可工作的，是默认工具链，例如：

- 语言运行时
- 常用依赖
- Git
- 测试工具
- 浏览器

这些工具的重要性不只是“能用”，而是让 Agent 能观察自己的工作结果，比如：

- 看日志
- 跑测试
- 截图页面
- 检查输出

于是就形成一个闭环：

> 写代码 → 运行 → 观察 → 修复 → 再运行

这也是很多 Agent 真正开始“像工程师”而不是“像聊天机器人”的关键点。

### 4. 记忆与搜索：让 Agent 有持续学习能力

模型的知识只来自两部分：

- 训练时写进权重的知识
- 当前上下文里注入的信息

除此之外，没有真正的长期记忆，也不会主动更新知识。

因此，如果希望 Agent 具备持续学习能力，就必须通过 Harness 把知识重新注入进上下文。

文章给出的方式很朴素，但也很实用：

- 维护记忆文件（比如 `AGENTS.md`）
- 在运行时写入新经验
- 在下一次启动时重新加载

这其实就是一种“文件式长期记忆”。

而对于模型无法预知的实时信息，还需要另一类能力：搜索与外部知识获取，例如：

- Web Search
- Context7 这类上下文查询工具
- MCP 提供的文档 / API 检索接口

本质上，这些能力的作用就是：

> 把模型“原本看不到的信息”拉进上下文。

### 5. 对抗 Context Rot：上下文管理工程化

文章很强调一个实际问题：Agent 一旦工作时间变长，上下文越来越长，模型表现往往会明显下降。

这就是所谓的 **Context Rot（上下文衰减）**：

- 信息越来越多
- 噪音占比越来越高
- 关键线索被淹没
- 推理质量开始不稳定

因此 Harness 的核心任务之一，就是把“上下文管理”工程化。

作者提到几个关键方法：

#### 压缩（Compaction）

当上下文快满时，不是继续无脑堆内容，而是：

- 总结对话
- 保留关键结论
- 把细节移出上下文

#### 工具输出卸载（Tool Output Offloading）

工具返回经常非常长，比如：

- 日志
- 测试输出
- 命令结果

真正有用的信息可能只占极小一部分。

更合理的策略是：

- 上下文里只保留开头 + 结尾 + 关键信号
- 完整输出写入文件系统
- 需要时再读取

#### Skills 与延迟加载

很多系统一开始就把大量工具说明、MCP 文档、规则描述全部塞进上下文，导致还没开始工作就已经污染严重。

文章认为更好的方式是：

- 先给最小必要信息
- 需要时再加载相关技能与说明

也就是所谓的渐进式披露或懒加载。

### 6. 长时程任务执行：让 Agent 真正把事情做完

文章指出，现在很多模型的真正问题，不是不会写代码，而是：

- 容易过早停止
- 不擅长拆解复杂任务
- 跨多个上下文窗口后开始不连贯

所以 Harness 的核心职责之一，是让工作能够跨时间、跨上下文持续推进。

作者提到几个关键机制：

#### 文件系统 + Git

把过程记下来，让状态外部化：

- 当前进展写进文件
- 改动历史交给 Git
- 新 Agent 可以快速接手

#### Ralph 循环

这是一个针对“做一半就停”的机制。

它的思路是：

- 拦截 Agent 的结束信号
- 给它一个更干净的上下文
- 让它继续朝目标推进

关键点在于：**上下文可以重置，但状态不能丢。**

#### 规划 + 自我验证

Agent 不仅要持续做，还要持续对。

所以还需要：

- 把任务拆成步骤并写进文件
- 每完成一步就跑测试 / 看日志 / 检查输出
- 如果失败，再把结果反馈回去继续修

于是形成稳定闭环：

> 执行 → 检查 → 反馈 → 修正

## Harness 与模型是共同进化的

文章并不把 Harness 和模型看成完全分离的东西，而是强调它们其实在共同演化。

今天很多 Agent 产品，比如 Claude Code、Codex，本质上都是模型与 Harness 共同优化的结果。

训练过程中，模型不只是学会生成文本，还会越来越擅长使用 Harness 提供的这些能力：

- 文件系统操作
- Bash 执行
- 任务规划
- 子 Agent 协作

于是形成一个反馈循环：

1. Harness 提供操作原语
2. 模型学习如何调用这些原语
3. 训练结果反过来塑造下一代模型
4. 模型在相同 Harness 下表现越来越强

但副作用也很明显：

- 模型可能对某个特定 Harness 过拟合
- 换到别的环境里，表现会下降

文章举了 Codex 文件编辑工具 `apply_patch` 的例子，说明模型在训练中习惯某种工作方式后，切换补丁逻辑就可能出问题。

作者还引用了 Terminal Bench 2.0 这类对比，想表达一个意思：

> **同一个模型，在不同 Harness 环境下，表现可能差非常多。**

这说明最终效果不只是“模型能力”的问题，而是“模型 × Harness 配合”的问题。

## 文章的最终判断

作者最后的态度并不是说“以后只剩 Harness 重要，模型不重要”，而是更平衡：

- 未来模型会越来越强
- 一部分今天放在 Harness 里的能力，未来可能被模型内化
- 但这不意味着 Harness 会失去价值

因为 Harness 的作用不只是补模型短板，更是：

- 设计任务执行系统
- 配置工作环境
- 组织工具能力
- 管理状态与验证闭环
- 让模型在实际任务里发挥最大效率

文章用了一个很形象的比喻：

- **Harness 是舞台与幕后控制系统**
- **Agent 是舞台上的演员**

演员再强，没有舞台、规则和协调系统，也发挥不出全部能力。

## 我的提炼

如果把这篇文章压缩成一句话，就是：

> **Prompt 工程关注“怎么跟模型说”，Harness 工程关注“怎么让模型真正把事做完”。**

它最值得记住的几个点是：

- Agent 的问题，很多不是模型智力问题，而是系统设计问题
- 文件系统、Git、Bash、沙箱、浏览器、搜索、记忆、验证闭环，这些都不是“配件”，而是 Agent 真正能工作的基础设施
- 上下文管理本身就是一门工程
- 同一个模型在不同 Harness 下，表现差异可能非常大
- 未来真正有竞争力的 AI 编程产品，比的未必只是模型，而是整套 Harness 设计能力

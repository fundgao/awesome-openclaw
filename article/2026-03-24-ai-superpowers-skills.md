# 给 AI 超能力？Superpowers 的设计与取舍

原文链接：https://kaochenlong.com/ai-superpowers-skills

## 一句话总结

这篇文章不是在教你安装或使用 Superpowers，而是在拆解它背后的设计理念：**AI 写代码最大的问题不是“不会”，而是“没纪律”**，所以需要一套强约束的工作流，把需求澄清、计划拆解、测试驱动、系统调试、结果验证、代码审查串成完整闭环。

## 核心观点

作者认为，当前 LLM 在编程任务中的常见问题，不是智力不够，而是缺少工程纪律，比如：

- 需求还没聊清楚就直接开工
- 还没验证就说“已经修好了”
- 没有先写测试就开始写业务代码
- debug 时靠猜，不做根因分析
- 收到 review 意见后直接迎合，而不是先验证再判断

Superpowers 这套规则的目标，就是把这些“人类工程师靠经验养成的习惯”，显式写成 AI 必须遵守的流程文档。

## Superpowers 是什么

Superpowers 本质上不是一个框架，也不是一个库，而是一组给 AI Agent 用的 **Skills 文档**。

它的核心形式是 15 个 Markdown 文件，每个 Skill 规定一种特定场景下的流程与约束。再配合一个 `SessionStart` hook，在每次会话开始时把 `using-superpowers` 注入进去，让 AI 一开始就知道有哪些规范可以调用。

作者把它理解成：**给 AI Agent 的员工手册**。

## 文中重点拆解的几个 Skill

### 1. brainstorming：先问清楚，再设计

这个 Skill 强制 AI 在开发前先通过“苏格拉底式提问”澄清需求，核心规则包括：

- 一次只问一个问题
- 能用选择题就不用开放题
- 不要一口气抛出一堆问题压垮用户

当 AI 初步理解需求后，还要写出 200~300 字设计方案，并分段向用户确认“这样对吗”。

作者认为这个设计很聪明，因为它避免 AI 一次性抛出完整方案、用户懒得认真看就直接点头的情况。

此外，AI 还必须提供 2~3 个可选方案，说明取舍，并遵守 **YAGNI（You Ain’t Gonna Need It）**，积极砍掉不必要的设计。

### 2. writing-plans：把计划写到“零上下文的人也能执行”

这个 Skill 对实施计划要求极严。

默认假设执行计划的人：

- 对项目毫无背景
- 判断力有限
- 不爱写测试

因此，计划必须被拆成 **2~5 分钟级别的原子步骤**。每一步都要写清：

- 文件路径
- 执行命令
- 预期输出
- 具体代码

不能出现“补上适当测试”这种模糊描述。

作者认为，这种方式虽然啰嗦，但能极大降低执行歧义，也让新的 AI session 或新成员能直接照单执行。

### 3. test-driven-development：没有失败测试，就不能写代码

这是文章着墨最重的部分之一。

Superpowers 对 TDD 的定义非常激进：

- **没有先写失败测试，就不能写生产代码**
- 如果已经先写了代码，再回头补测试，正确做法不是保留代码参考，而是**删掉重来**

整个流程严格遵循：

1. 先写一个会失败的测试
2. 跑一次，确认它真的失败
3. 再写最小代码让测试通过
4. 再跑一次确认通过
5. 最后再重构

文章特别提到，Superpowers 还会提前堵住各种常见借口，比如：

- “这太简单了，不需要测试”
- “我已经手动测过了”
- “先写代码再补测试效果一样”
- “删掉几小时代码太浪费”

作者认为，这种“预判借口并逐条反驳”的写法很少见，但对 AI 特别有效。

### 4. systematic-debugging：修 bug 前先找根因，不准乱猜

这个 Skill 把调试过程拆成四个阶段：

1. **根因调查**：读错误信息、稳定复现、检查最近改动、在边界打日志
2. **模式分析**：找正常案例做对比，看自己哪里不一样
3. **假说与验证**：提出明确假说，一次只改一个变量验证
4. **实施修复**：先写复现 bug 的测试，再修，再验证

其中一个关键要求是：

- **如果连续修了 3 次还没好，就别再硬修了，要怀疑架构本身有问题**

作者认为这是很实用的限制，因为很多 AI 和人类开发者一样，debug 时容易进入“猜一下、试一下、再试一下”的鬼打墙模式。

### 5. verification-before-completion：没有验证证据，不准说“完成了”

这个 Skill 针对 AI 的另一个常见坏习惯：**还没验证就宣布完成**。

核心原则很直白：

- 在说“完成了”“修好了”“测试通过了”之前，必须先执行对应验证命令
- 必须读完整输出
- 必须确认 exit code 和结果都支持这个结论

不能说：

- “应该可以”
- “看起来没问题”
- “我有信心这次好了”

作者很认同这个原则，因为它要求 AI **先拿证据，再下结论**。

### 6. subagent-driven-development：两阶段审查，先看“是不是要的”，再看“写得好不好”

这个 Skill 设计了一个双重审查流程：

- 第一阶段：**规格符合性审查**
  - 检查是否严格按需求完成，不多做也不少做
- 第二阶段：**代码质量审查**
  - 在规格过关后，再看架构、可维护性、测试覆盖率等

作者认为这个顺序很合理，因为“代码很好，但方向错了”是很常见的问题。先确认做的是对的，再打磨做得好不好，效率更高。

### 7. receiving-code-review：收到 review 不要表演式认同

这部分很有意思。

Skill 明确禁止 AI 在收到 review 后先来一段无意义的情绪性反馈，例如：

- “你说得对！”
- “好建议！”
- “非常棒的反馈！”

它要求的正确流程是：

1. 先复述技术要求，确认自己理解正确
2. 验证建议是否适用于当前代码库
3. 判断建议是否真的合理
4. 再决定接受、追问或技术性反驳

作者认为，这能避免 AI 出于“显得配合”的冲动，盲目接受不合适的 review 意见。

### 8. writing-skills：写 Skill 本身，也用 TDD 思维

这个 Skill 的设计理念是：

- **写 Skill，本质上也是一种 TDD**

流程不是凭空写规则，而是：

1. 先设计压力测试场景
2. 观察 AI 在没有 Skill 时怎么失败、怎么绕规则
3. 再写 Skill 去堵这些真实失败模式
4. 然后重新测试，看 Skill 是否真的起效

作者很认同这个方法，因为它强调 **规则不是拍脑袋想出来的，而是从真实失败中提炼出来的**。

## 文章认为 Superpowers 最大的价值

作者并不只是夸它，而是强调它真正有价值的地方在于：  
它把“AI 写软件时应该遵守什么工程流程”这件事，系统化地思考了一遍。

几个特别值得借鉴的设计点：

- **强制语气，而不是建议语气**
  - 不是“你应该”，而是“你必须”
- **主动预判借口**
  - 不是讲大道理，而是堵漏洞
- **每一步都要求验证**
  - 防止“以为好了，其实没好”
- **任务原子化**
  - 让新 subagent 也能接手，不依赖长期上下文

## 作者指出的限制

文章也没有把 Superpowers 神化，反而讲得很实在：

### 1. 仍然依赖 AI 自律
这些规则本质还是文档，不是强制执行的程序。  
AI 可以遵守，也可以跳过。

### 2. 成本高
如果每个任务都开实现 agent、规格审查 agent、质量审查 agent，token 消耗会很快。

### 3. 对大型任务会变笨重
如果任务拆成上百个小步骤，计划很难维护，一旦中途改方向会非常痛苦。

### 4. 默认开发环境已经比较规范
例如项目已经有测试命令、build 命令、Git 流程，否则很多 Skill 会失效或需要重写。

### 5. 自己也可能过度设计
文章最后点出一个有意思的悖论：  
Superpowers 一边强调不要过度设计，一边自己又可能是某种“防止过度设计的过度设计”。

## 作者给出的使用判断

文章的态度不是“所有人都该用”，而是更细分地看场景。

更适合：

- 长期维护项目
- 对质量要求高的项目
- 团队刚开始让 AI 参与开发，缺少稳定流程

不太适合：

- 快速试错原型
- 一次性脚本
- 目标就是快速 vibe coding 的场景

因为这套流程会显著增加设计、验证、审查成本。

## 我的提炼

如果把这篇文章压缩成一句话，就是：

> Superpowers 不是在提升 AI 的智力，而是在补 AI 的工程纪律。

它最有启发的点，不是某一个 Skill 的细节，而是它提醒我们：

- AI 容易绕流程
- AI 容易过早下结论
- AI 容易跳过验证
- AI 需要的不是“更自由”，而是“更明确的约束”

即使你不照搬整套 Superpowers，只吸收其中几条原则，比如：

- 先澄清需求再动手
- 先测失败再写代码
- 先查根因再修 bug
- 先验证结果再宣称完成

也已经能显著提高 AI 辅助开发的可靠性。

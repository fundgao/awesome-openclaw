# Anthropic 说：别等下一代模型了，先把 Harness 搭起来

原文链接：https://mp.weixin.qq.com/s/iTLZQHw9Dkt6AsDumZZJNQ

## 一句话总结

这篇文章的核心观点是：**现阶段很多 Agent 系统的上限，不是被模型智力卡住，而是被模型外的 Harness（运行壳、约束、反馈与验收机制）卡住；与其一味等更强模型，不如先把 Harness 工程做好。**

## 核心观点

作者基于 Anthropic、OpenAI、Stripe、Cursor、LangChain 等公开案例，强调一个越来越清晰的行业判断：

> **同一个模型，换一层更好的壳，效果可能比“换下一代模型”提升更大。**

这里的 Harness，不只是 prompt 或 tools，而是一整套让 Agent 能持续、稳定、可验证地工作的系统，包括：

- 任务拆解与执行编排
- 规则、约束与边界
- 验收机制与评估 Agent
- CI、linter、测试与反馈回路
- 上下文管理与生命周期控制

文章的重点不在“Agent 会不会写代码”，而在：

- **怎么让它持续写对**
- **怎么避免它重复犯错**
- **怎么让它在长任务中不跑偏、不自我感动、不草率宣布完成**

## 为什么现在要重视 Harness

### 1. 只改壳，不改模型，结果就能大幅变化

文中列举了几组非常有冲击力的数据：

- **Nate B Jones**：同一模型，仅改变模型外运行方式，编程成功率从 **42% 提升到 78%**
- **LangChain**：同一模型仅改 Harness，在 Terminal Bench 2.0 上从 **52.8% 提升到 66.5%**，排名从三十名开外直接进前五
- **Anthropic**：同一句需求、同一模型，solo 模式跑几十分钟产出“看起来像样但核心功能坏掉”的结果；换成带规划与验收的 harness 模式，虽然更贵更久，但能做出**真正可玩的成品**

作者借这些例子说明：

> **很多时候瓶颈不在模型本身，而在模型外面的工程系统。**

### 2. 模型不会可靠地评价自己

文章特别强调 Anthropic 揭示的一个底层问题：

> **模型很不擅长客观评估自己刚写出来的东西到底行不行。**

它往往会：

- 觉得“差不多就行”
- 过早宣布完成
- 只做表面测试
- 忽略边界情况
- 明明核心功能是坏的，却仍然自信给出“完成了”的结论

这也是为什么仅靠一个“能生成”的 Agent 不够，还需要**独立的评估与验收环节**。

## Harness Engineering 是什么

文章沿着三代演化去理解这个概念：

### 第一代：Prompt Engineering

关注点是：**怎么写好一条指令。**

典型做法是 few-shot、角色扮演、chain-of-thought 等，核心仍然是单轮输入输出优化。

### 第二代：Context Engineering

关注点变成：**怎么给模型准备正确的上下文。**

例如：

- 相关文件
- 历史对话
- 工具定义
- 检索结果
- 项目背景

也就是让模型在每次决策时看到“该看到的东西”。

### 第三代：Harness Engineering

再往上一层，关注的是：**怎么为 Agent 搭建一整套工作环境与控制系统。**

不只是给它信息，而是给它：

- 规则
- 边界
- 工具链
- 生命周期管理
- 反馈回路
- 自动化验收
- 错误后的纠偏机制

作者用一个很直观的类比：

- Prompt Engineering：像写好一封邮件
- Context Engineering：像把相关附件都带上
- Harness Engineering：像把整套工作环境、流程和检查机制都搭起来

## Hashimoto 的定义：让错误不再重复发生

文章回到这个词最早的传播者之一 Mitchell Hashimoto。

他的定义非常实用：

> 每次发现 Agent 犯错，不是骂它、不是重试，而是把这个错误抽象成一个工程问题，设计一个解决方案，让它以后别再犯同样的错。

这也是很多团队现在维护 `AGENTS.md`、规则文件、lint 规则、执行模板的根本思路：

- 规则不是拍脑袋写的
- 规则应该来自真实失败案例
- 每条规则背后，都对应一个“曾经踩过的坑”

## OpenAI：百万行代码背后的结论

文章引用了 OpenAI Codex 团队那组很有代表性的实验：

- 从空仓库开始
- 5 个月
- 约 100 万行代码
- 约 1500 个 PR
- 人类一行代码没写

但真正让人警醒的不是“AI 写了多少”，而是团队最后的结论：

> **Agent 不难，Harness 才难。**

文章提炼了 OpenAI 这类团队的几条共识：

- 仓库是 Agent 的唯一事实源
- 代码不仅要给人读，也要给 Agent 读
- 架构约束不能只靠 prompt，要靠 **linter / 结构测试 / 机械化执行**
- Agent 的自主权要逐步放开，而不是一开始就全权放手
- 如果一个 PR 总要人工大改才能合并，问题更多在 **Harness 不够好**，而不是单次模型输出不够好

换句话说，工程师越来越像在做这些事：

- 写规则
- 写反馈回路
- 写验收机制
- 写架构边界

而不是亲自敲每一行业务代码。

## Stripe / Cursor：大规模并行下，Harness 更像“操作系统”

### Stripe 的做法

Stripe 内部的 Minions 系统体现了比较成熟的无人值守并行范式。

文章提到它的几个关键点：

- 用 Blueprint 把工作流拆成 **确定性节点** 和 **Agentic 节点**
- 固定流程（如 linter、推送变更）尽量不交给模型
- 需要判断和创造性的环节，才交给 Agent
- CI 最多只允许两轮：失败一次自动修，再失败就交回人类，不让 Agent 无限重试
- 虽然平台上有数百个 MCP 工具，但分配给单个 Agent 的工具是精简过的

这背后的判断非常重要：

> **更多工具 ≠ 更强表现。**

### Cursor 的教训

Cursor 在做长期自动编码实验时，踩过很多典型的 Harness 坑：

- 单 Agent 扛不住复杂任务
- 多 Agent 共享状态文件时会互相打架
- 角色分工过死又会变僵化
- 模糊目标在高并发下会被不断放大

最终他们走向了更清晰的 Planner-Worker 结构。

作者借这个案例强调：

> **Harness 不只是“给模型加工具”，而是控制并发、管理边界、减少状态污染的系统设计。**

## Anthropic 的关键突破：拆分 Generator 与 Evaluator

文章最有价值的部分之一，是对 Anthropic 做法的拆解。

Anthropic 发现一个强现实问题：

- 生成 Agent 可以写东西
- 但让它自己验收自己，往往不靠谱

于是他们借鉴了“生成 / 评估分离”的思路：

### 1. Planner

先把一句模糊需求展开成更完整的产品规格或执行计划。

### 2. Generator

负责按计划实现功能。

### 3. Evaluator

不看“感觉”，而是真去：

- 用 Playwright 点页面
- 调接口
- 看数据库状态
- 检查核心交互
- 像 QA 一样做端到端验收

文章特别强调了一个判断：

> **把 evaluator 调教得足够严格，比让 generator 学会自我批评更现实。**

这就是为什么“多一个独立验收 Agent”有时会显著改善结果质量。

## Harness 的本质：把“想当然”变成“有证据”

作者反复强调，Harness 的价值并不是给 Agent 增加花哨能力，而是把整个工作流从“想当然”变成“有反馈、有证据、有边界”。

它主要解决这些问题：

### 1. 过早宣布完成

Agent 没有严格验证就容易自我庆祝。

### 2. 任务做一半跑偏

长任务中上下文会膨胀，方向会漂移。

### 3. 质量无法稳定复现

如果没有固定的测试、lint、评估流程，同样模型今天能成，明天也可能翻车。

### 4. 错误会重复发生

没有把失败固化成规则，Agent 会一遍遍踩同样的坑。

### 5. 并行度上来后，系统性错误会被放大

一个模糊指令、一个有歧义的规则，在数百个并发 Agent 下会被成倍扩散。

## 也不是所有人都完全认同

文章没有只站在“Harness 万能论”一边，也提到了一种相反观点。

比如 Noam Brown 认为：

> **Harness 在某种程度上像“拐杖”，随着模型能力增强，其中一部分会被淘汰。**

他的理由是：

- 过去很多复杂编排，其实是在补旧模型不会推理的缺陷
- 一旦更强的推理模型出现，某些多层包装可能反而会成为负担

文章也引用了一些现实数据来提醒不要过度乐观：

- 自动化评分并不等于真实可合并质量
- 维护者对 AI PR 的接受率可能远低于 benchmark 看起来的水平
- 有些“Agent 已经能做 50 分钟任务”的结论，落到真实仓库未必成立

也就是说：

> **Harness 很重要，但不能用华丽的工程壳掩盖真实质量问题。**

## 为什么说 Noam Brown 只对了一半

作者最后的判断很有意思：

- **有些 Harness 会随着模型升级而失效或变得多余**
- **但 Harness 本身不会消失，它只会迁移到更高阶的问题上**

例如：

- 以前模型弱，要靠 sprint 拆分和频繁 context reset 才能跑长任务
- 模型变强后，这些低阶约束可能可以删掉
- 但当你开始让它一次自主工作 2 小时、4 小时、6 小时时，又会冒出新的问题
- 这时你仍然需要新的验收机制、反馈回路和边界控制

所以更准确的说法不是“Harness 会被模型淘汰”，而是：

> **旧 Harness 会过时，但新的 Harness 需求会继续出现。**

## 文章最后的落点

作者的最终结论很明确：

> **写代码正在快速变便宜，而设计那套让 Agent 持续、稳定、高质量写代码的系统，才是真正稀缺和昂贵的能力。**

这意味着工程师角色正在发生迁移：

- 从“亲自写每一行代码”
- 转向“设计环境、规则、反馈和验收系统”

而且这套系统还不是一劳永逸的。每一代模型出来，都要重新判断：

- 哪些约束已经没必要
- 哪些验收仍然必须保留
- 哪些新的组合空间被打开

## 我的提炼

如果把这篇文章压成一句更实用的话，那就是：

> **不要只盯着模型参数表，真正决定 Agent 能不能稳定上生产的，往往是模型外那套被你认真设计过的工程壳。**

对普通团队来说，比起空等“下一代模型拯救一切”，更现实的优先级通常是：

- 先把仓库文档整理好
- 先把测试和验收补起来
- 先把规则写清楚并机械化执行
- 先把 Agent 的边界、上下文和反馈回路设计明白

这才是 Harness Engineering 真正落地的起点。
